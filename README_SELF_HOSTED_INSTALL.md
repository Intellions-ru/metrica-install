# Как установить Интеллион Метрика у себя

Это основной документ для клиента, который купил `Интеллион Метрика` и хочет поставить систему на свой сервер в `self-hosted` режиме.

Здесь нет внутренней кухни разработки. Только рабочий путь:

1. подготовить сервер;
2. запустить установку;
3. дождаться автоподключения `nginx` или при необходимости применить готовый proxy-шаблон;
4. открыть панель;
5. активировать владельца;
6. подключить свой сайт и проверить первые данные.

Дополнительный документ для ручной приемки:

- `README_SELF_HOSTED_MANUAL_TESTS.md`

## Быстрый старт

Если вы уже на сервере и готовы запускать установку, используйте этот путь.

### Самый простой вариант

Ничего подставлять в команду не нужно.

Просто запустите installer, а нужные значения он спросит сам:

```bash
curl -fsSL https://raw.githubusercontent.com/Intellions-ru/metrica-install/v0.2.9/install_metrica.sh | sudo bash
```

Installer по очереди спросит:

- домен;
- имя установки;
- email владельца;
- email для TLS-уведомлений;
- нужно ли настраивать MAX прямо сейчас.

По умолчанию installer сам выбирает основной продуктовый режим:

- `attach-path`
- путь `/metrica`

То есть отдельный первый вопрос про режим публикации больше не нужен в обычном сценарии.

Это основной путь для живой ручной установки на сервере.

### Быстрый вариант через install bundle

```bash
curl -fLO https://github.com/Intellions-ru/metrica-install/releases/download/v0.2.9/intellion-metrica-install-bundle-v0.2.9.tar.gz
tar -xzf intellion-metrica-install-bundle-v0.2.9.tar.gz
cd intellion-metrica-install-bundle-v0.2.9
sudo bash ./scripts/install_metrica.sh
```

### Вариант с заранее подставленными значениями

Если вам удобнее не отвечать на вопросы, а сразу передать параметры в команду, используйте такой вариант:

```bash
curl -fsSL https://raw.githubusercontent.com/Intellions-ru/metrica-install/v0.2.9/install_metrica.sh | sudo bash -s -- \
  --publish-mode attach-path \
  --domain YOUR_DOMAIN \
  --entry-path /metrica \
  --installation-name "YOUR INSTALLATION NAME" \
  --owner-email YOUR_EMAIL
```

Что нужно заменить в такой команде:

- `YOUR_DOMAIN` — домен клиента, где будет открываться Метрика
- `YOUR INSTALLATION NAME` — понятное имя установки
- `YOUR_EMAIL` — почта первого владельца

Во время интерактивной установки может появиться вопрос:

```text
Введите почту для TLS-уведомлений [YOUR_EMAIL]:
```

Если почта для TLS-уведомлений должна быть той же самой, просто нажмите `Enter`.

Также установщик может спросить:

```text
Настроить MAX-бота сейчас? [y/N]:
```

Отвечайте `y` только если токен MAX уже готов.

Если вы ответили `y`, а потом передумали, на шаге

```text
Введите токен MAX-бота (Enter чтобы пропустить):
```

можно просто нажать `Enter`. Установка продолжится без настройки MAX-бота.

Пример для реального клиента:

```bash
sudo bash ./scripts/install_metrica.sh \
  --publish-mode attach-path \
  --domain beauty-doc.pro \
  --entry-path /metrica \
  --installation-name "Beauty Doc Analytics" \
  --owner-email owner@beauty-doc.pro
```

Что должно получиться:

- installer завершится без критической ошибки;
- на сервере появится `owner-activation.txt`;
- если на этом же сервере уже работает host `nginx` или `dockerized nginx` с bind-mounted конфигом и найден один подходящий `server_name`, путь `/metrica` будет подключен автоматически;
- это касается и случая, когда `dockerized nginx` смонтирован не директорией, а одним файлом `nginx.conf`;
- в остальных случаях установщик сохранит готовый proxy-шаблон и прямо скажет, что осталось применить вручную.

## Что это за продукт

`Интеллион Метрика` — self-hosted система аналитики сайта.

После установки вы получаете:

- панель Метрики;
- первого владельца установки;
- добавление и настройку своих сайтов;
- безопасный прием событий и согласий;
- цели, сводки и рабочий управленческий контур.

Метрика ставится на ваш Linux-сервер и публикуется:

- либо по пути вида `site.ru/metrica`;
- либо на отдельном поддомене;
- либо, при необходимости, как отдельный standalone-контур.

Основной клиентский сценарий: тот же сервер и путь `/metrica`.

## Что нужно до начала установки

Минимально нужно:

- Linux-сервер;
- SSH-доступ;
- `root` или `sudo`;
- домен сайта или поддомен для панели;
- почта первого владельца;
- сайт, который вы будете подключать к Метрике.

По серверу:

- память: минимум `2 GB`, желательно `4 GB` и выше;
- диск: минимум `8 GB`, желательно `15 GB` и выше;
- исходящий доступ в интернет для загрузки образов.

`Docker` ставить вручную не обязательно. Установщик проверит его сам и, если нужно, установит автоматически.

## Какой режим публикации выбрать

### Основной режим: тот же сервер и путь `/metrica`

Пример:

- `https://example.com/metrica`
- `https://beauty-doc.pro/metrica`

Это основной продуктовый путь.

Что это значит:

- основной сайт продолжает работать как обычно;
- существующий `nginx`, `caddy` или `apache` остается главным внешним веб-сервером;
- Метрика запускается на внутренних портах;
- наружу вы публикуете ее через готовый proxy-шаблон, который создаст установщик.

### Второй режим: отдельный поддомен

Пример:

- `https://analytics.example.com`

Этот вариант удобен, если:

- у вас уже есть поддомен;
- вы хотите отделить панель от основного сайта;
- вам так проще по инфраструктуре.

### Дополнительный режим: standalone

Этот вариант нужен не всем.

Он подходит, если:

- сервер пустой;
- вы хотите, чтобы Метрика сама поднимала внешний веб-сервер;
- вы сознательно выделяете под нее отдельный внешний контур.

Для большинства клиентов основной выбор такой:

1. `attach-path`
2. `attach-subdomain`
3. `standalone`

## Что важно понять до запуска

В режимах `attach-path` и `attach-subdomain` установщик:

- не забирает `80/443` себе;
- не ломает существующий сайт сам по себе;
- не переписывает боевой reverse proxy вслепую.

Что он делает:

- ставит сервисы Метрики;
- поднимает их на внутренних портах;
- в основном сценарии `attach-path` пытается безопасно подключить путь `/metrica` в `nginx` автоматически;
- если безопасное автоподключение невозможно, создает готовые proxy-шаблоны и показывает, какой конфиг нужно подключить вручную.

Это и есть безопасная модель для сервера, где уже работает сайт.

## Подготовка сервера

### Шаг 1. Подключитесь к серверу

Пример:

```bash
ssh root@<IP_СЕРВЕРА>
```

Если вы работаете не под `root`, нужен пользователь с `sudo`.

### Шаг 2. Из какой папки запускать установку

Запускать установку из папки сайта не нужно.

Можно работать из любой удобной папки, например:

- `/root`
- `/home/<ваш_пользователь>`
- `/tmp`

Сама система все равно развернется в свой рабочий каталог:

```bash
/opt/intellion-metrica
```

### Шаг 3. Проверьте, что домен смотрит на ваш сервер

Если вы хотите открыть Метрику как:

- `example.com/metrica`

то домен `example.com` должен указывать на тот сервер, где вы ставите Метрику.

Если вы хотите открыть Метрику как:

- `analytics.example.com`

то на ваш сервер должен указывать именно этот поддомен.

Примеры:

- `example.com -> 203.0.113.10`
- `analytics.example.com -> 203.0.113.10`

Если домен смотрит не на тот сервер, установка может пройти, но панель снаружи не откроется.

Быстрая проверка:

```bash
nslookup example.com
```

или:

```bash
ping example.com
```

### Шаг 4. Заранее определите 3 значения

Перед запуском установщика нужно просто заранее решить:

- по какому адресу будет открываться Метрика;
- как будет называться эта установка;
- какая почта станет почтой первого владельца.

Пример:

- адрес панели: `example.com/metrica`
- имя установки: `Example Analytics`
- почта владельца: `owner@example.com`

Что это значит:

- `адрес панели` — адрес, по которому вы будете открывать Метрику;
- `имя установки` — понятное имя вашей установки в самой панели;
- `почта владельца` — email человека, который первым активирует доступ.

Важно:

- автоматическое письмо на эту почту сразу после установки не отправляется;
- после установки на сервере появится одноразовая ссылка активации;
- по этой ссылке владелец сам задаст пароль.

## Основная команда установки

### Вариант 1. Из install bundle

Сначала скачайте install bundle на сервер:

```bash
curl -fLO https://github.com/Intellions-ru/metrica-install/releases/download/v0.2.9/intellion-metrica-install-bundle-v0.2.9.tar.gz
```

Этот bundle уже включает:

- установщик;
- buyer-facing инструкции;
- шаблоны proxy;
- product images для Метрики.

То есть отдельный вход в Docker registry для обычного старта не нужен.

Распакуйте архив и перейдите в каталог:

```bash
tar -xzf intellion-metrica-install-bundle-v0.2.9.tar.gz
cd intellion-metrica-install-bundle-v0.2.9
```

Запустите установку:

```bash
sudo bash ./scripts/install_metrica.sh \
  --publish-mode attach-path \
  --domain example.com \
  --entry-path /metrica \
  --installation-name "Example Analytics" \
  --owner-email owner@example.com
```

### Вариант 2. По прямой ссылке на установщик

```bash
curl -fsSL https://raw.githubusercontent.com/Intellions-ru/metrica-install/v0.2.9/install_metrica.sh | sudo bash -s -- \
  --publish-mode attach-path \
  --domain example.com \
  --entry-path /metrica \
  --installation-name "Example Analytics" \
  --owner-email owner@example.com
```

### Нужен ли Git

Нет. Для обычной установки клиенту `git clone` не нужен.

Нормальный путь:

- скачать install bundle;
- либо сразу запустить установщик по прямой ссылке.

### Что означают параметры

- `--publish-mode attach-path` — основной режим, Метрика будет открываться по пути вида `/metrica`
- `--domain example.com` — домен, на котором будет работать панель
- `--entry-path /metrica` — путь панели для режима `attach-path`
- `--installation-name` — понятное имя установки
- `--owner-email` — почта первого владельца
- `--acme-email` — почта для TLS-уведомлений; если она совпадает с `--owner-email`, достаточно нажать `Enter` на интерактивном шаге
- `--max-bot-token` — токен MAX-бота; параметр необязательный, если токена сейчас нет, MAX можно подключить позже
- `--no-auto-attach-proxy` — отключает безопасную автоматическую публикацию `/metrica` в `nginx` и оставляет только ручной proxy-шаблон

### Пример для поддомена

```bash
curl -fsSL https://raw.githubusercontent.com/Intellions-ru/metrica-install/v0.2.9/install_metrica.sh | sudo bash -s -- \
  --publish-mode attach-subdomain \
  --domain analytics.example.com \
  --installation-name "Example Analytics" \
  --owner-email owner@example.com
```

## Что происходит во время установки

Во время установки система:

1. проверяет сервер;
2. проверяет или ставит `Docker`;
3. готовит рабочий каталог;
4. загружает и запускает сервисы;
5. применяет изменения структуры базы;
6. создает владельца в режиме ожидания активации;
7. создает install report;
8. генерирует proxy-шаблоны для внешней публикации.

Для режимов `attach-path` и `attach-subdomain` это значит:

- Метрика уже установлена;
- но внешний веб-сервер еще нужно один раз подключить к ней по готовому шаблону.

## Что пользователь получает после установки

После установки вы увидите:

- адрес панели;
- почту владельца;
- путь к install log;
- путь к install summary;
- путь к файлу активации владельца;
- путь к proxy-шаблонам.

Самое важное:

- install log:  
  `/opt/intellion-metrica/logs/install-<timestamp>.log`
- install summary:  
  `/opt/intellion-metrica/artifacts/install/<timestamp>-one-command-install/install-summary.json`
- файл активации владельца:  
  `/opt/intellion-metrica/state/owner-activation.txt`
- proxy-шаблоны:  
  `/opt/intellion-metrica/runtime/proxy`

## Как подключить Метрику к существующему reverse proxy

Это fallback-шаг для режимов `attach-path` и `attach-subdomain`, если автоподключение не было выполнено.

Для `attach-path` на том же сервере с `nginx` установщик сначала пытается сделать все сам:

- находит один подходящий `server_name` для вашего домена;
- подключает managed snippet;
- проверяет `nginx -t`;
- делает `reload nginx`;
- если на любом шаге что-то не так, откатывает изменение и оставляет только готовый шаблон для ручного подключения.

Ручное подключение нужно, если:

- у вас не `nginx`, а `caddy` или другой reverse proxy;
- `nginx`-конфиг слишком нестандартный;
- на сервере несколько конфликтующих `server_name` для одного домена;
- Метрика стоит не на том же сервере, что и боевой сайт;
- вы сознательно отключили автоподключение.

### Если у вас `nginx`

Смотрите файлы:

- `/opt/intellion-metrica/runtime/proxy/nginx/attach-path.conf`
- `/opt/intellion-metrica/runtime/proxy/nginx/attach-subdomain.conf`

Что делать:

1. откройте нужный файл;
2. вставьте его в ваш рабочий `nginx`-конфиг;
3. проверьте конфигурацию:

```bash
nginx -t
```

4. перезагрузите `nginx`:

```bash
systemctl reload nginx
```

### Если у вас `caddy`

Смотрите файлы:

- `/opt/intellion-metrica/runtime/proxy/caddy/attach-path.Caddyfile`
- `/opt/intellion-metrica/runtime/proxy/caddy/attach-subdomain.Caddyfile`

Что делать:

1. вставьте нужный блок в рабочий `Caddyfile`;
2. проверьте конфиг:

```bash
caddy validate --config /etc/caddy/Caddyfile
```

3. перезагрузите `caddy`:

```bash
systemctl reload caddy
```

### Если у вас `apache`

На первом этапе лучше использовать `nginx` или `caddy`.

Если у вас уже работает `apache`, безопаснее сначала согласовать конкретный reverse-proxy блок с Intellions.

## Первый вход владельца

### Шаг 1. Откройте файл активации

```bash
cat /opt/intellion-metrica/state/owner-activation.txt
```

Внутри будет одноразовая ссылка.

### Шаг 2. Откройте ссылку в браузере

Владелец открывает ссылку и задает пароль.

Что должно получиться:

- ссылка открывается;
- пароль задается без ошибки;
- после активации можно войти в панель.

### Если ссылка активации истекла

На первом этапе это лучше перевыпускать через Intellions или операторский сценарий.

### Если доступ утерян

Сначала проверьте:

- сохранился ли `owner-activation.txt`;
- не активирован ли владелец уже ранее;
- есть ли доступ к серверу и install log.

Если нет уверенного способа восстановить доступ самостоятельно, лучше написать в Intellions.

## Как подключить свой сайт к Метрике

### Шаг 1. Войдите в панель

После активации откройте адрес панели:

- `https://example.com/metrica`
- или `https://analytics.example.com`

### Шаг 2. Добавьте сайт

В панели откройте:

- `Управление`
- `Сайты`

Заполните минимум:

- код сайта;
- название;
- основной домен;
- разрешенные домены.

### Шаг 3. Сохраните настройки сайта

После сохранения панель подготовит данные для безопасного приема событий.

### Шаг 4. Подключите маршрут сайта к Метрике

Для сайта нужно направить клиентские события и согласия в Метрику.

Базовая схема:

- маршрут событий;
- маршрут согласий;
- безопасный прием через подписанный proxy-путь.

В обычной установке это делается через ваш сайт и его серверную конфигурацию. Если вы подключаете первый сайт самостоятельно и не уверены в схеме, лучше пройти этот шаг по инструкции handoff или с нашей помощью.

### Шаг 5. Проверьте, что данные пошли

После подключения:

- откройте сайт;
- перейдите по нескольким страницам;
- отправьте форму или нажмите на основную кнопку;
- вернитесь в Метрику и откройте `Обзор`.

Что должно получиться:

- сайт добавлен;
- события начинают появляться;
- согласия фиксируются, если они включены;
- цели можно настроить и проверить.

## Что проверить после подключения

Короткий чек-лист:

- панель открывается;
- владелец активирован;
- вход работает;
- сайт добавлен;
- события поступают;
- согласия фиксируются;
- цели создаются;
- отчеты работают.

## Частые ошибки и что делать

### Панель не открывается

Проверьте:

- домен действительно указывает на ваш сервер;
- installer не сообщил, что автоподключение proxy было пропущено или откатилось;
- если автоподключения не было, нужный proxy-шаблон подключен в `nginx` или `caddy`;
- конфиг proxy перезагружен без ошибки;
- контейнеры Метрики запущены.

### Владелец не активируется

Проверьте:

- используется актуальная ссылка из `owner-activation.txt`;
- пароль достаточно длинный;
- время на сервере выставлено правильно.

### Данные с сайта не идут

Проверьте:

- сайт создан в панели;
- код сайта указан верно;
- домен сайта добавлен в разрешенные;
- маршрут событий действительно подключен к Метрике;
- вы сделали тестовые действия после подключения.

### Proxy уже есть, но панель все равно не открывается

Проверьте:

- вы взяли шаблон именно для нужного режима: `attach-path` или `attach-subdomain`;
- внешний домен и внутренний порт не были изменены вручную с ошибкой;
- после изменения proxy был выполнен reload.

## Что входит в поддержку и когда обращаться в Intellions

Самостоятельно обычно можно:

- запустить установку;
- подключить готовый proxy-шаблон;
- активировать владельца;
- добавить сайт;
- выполнить первую ручную проверку.

Лучше писать в Intellions, если:

- не открывается панель после корректного proxy-подключения;
- не удается активировать владельца;
- не получается подключить сайт к collect/consent маршрутам;
- нужна доработка схемы под нестандартную инфраструктуру;
- нужен отдельный поддомен, перенос, перевыпуск доступа или handoff.

## Что уже подтверждено на практике

Подтверждено ручными smoke-прогонами:

- `attach-path`
- `attach-subdomain`

Проверено:

- открытие UI;
- owner activation;
- вход;
- admin surface;
- prefixed/static assets;
- `collect`;
- `consent`;
- signed ingest через product proxy.

Что еще полезно сделать отдельно перед клиентским handoff:

- прогон на живом внешнем домене клиента;
- финальную проверку конкретного боевого `nginx` или `caddy` конфига.

## Как удалить Метрику безопасно

Для удаления есть отдельный script:

```bash
sudo bash /opt/intellion-metrica/scripts/uninstall_metrica.sh --yes
```

Это безопасный режим удаления. Он:

- останавливает и удаляет контейнеры Метрики;
- удаляет только те `nginx`-вставки, которые были созданы самим installer;
- отключает managed systemd timer для MAX digest, если он ставился;
- сохраняет данные и сам каталог установки.

Если вы хотите поставить на этот же сервер новую установку Метрики с нуля, используйте именно `--purge-all`, а не только базовый safe-stop режим.

Если нужно убрать Метрику полностью, но сначала сделать backup:

```bash
sudo bash /opt/intellion-metrica/scripts/uninstall_metrica.sh --yes --purge-all
```

Этот режим:

- создает backup перед удалением;
- пытается сделать `pg_dump`, если база еще работает;
- архивирует каталог установки;
- удаляет контейнеры, volumes и install dir;
- удаляет только auto-managed proxy/snippet, но не трогает чужие ручные правки в `nginx`.

Если нужно убрать еще и product images:

```bash
sudo bash /opt/intellion-metrica/scripts/uninstall_metrica.sh --yes --purge-all --remove-images
```

Что важно:

- script не должен удалять чужие сайты или произвольные конфиги;
- он убирает только те ресурсы, которые относятся к установленной Метрике;
- если proxy подключался вручную вне managed installer path, такие ручные изменения нужно проверять и убирать отдельно.
